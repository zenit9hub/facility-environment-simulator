# TDD_개발로드맵_학생가이드

## 1. 문서 목적
이 문서는 설비환경시뮬레이터 프로젝트를 학생이 스스로 단계적으로 구현할 수 있도록 TDD 기반 실습 순서를 안내한다.

## 2. 이 프로젝트의 현재 위치
현재 프로젝트는 아래 단계까지 완료된 상태다.

### 2.1 완료된 준비 단계
- TypeScript 기반 프로젝트 구조 생성
- Vitest 기반 단위 테스트/통합 테스트 환경 구성
- MQTT 테스트 하네스 구성
- `npm run verify` 완료 게이트 구성
- 온도 계산 1차 규칙 구현
- MQTT 제어 메시지 수신 및 정기 발행 기본 흐름 구현
- 실행 엔트리포인트와 런타임 설정 로더 구현

즉, 학생 입장에서는 완전히 빈 프로젝트에서 시작하는 것이 아니라, TDD 개발 레일이 먼저 깔린 상태에서 다음 기능 Slice들을 이어서 구현하는 구조다.

## 3. 전체 개발 로드맵
시뮬레이터 MVP 기준으로는 크게 5단계, Node-RED Flow 연동 준비까지 포함하면 6단계로 본다.
운영 안정화까지 포함하면 8단계로 확장할 수 있다.

### 3.1 0단계. 개발 레일 이해
목표:
- 프로젝트 구조와 테스트 실행 방법을 이해한다.
- 설계 문서, README, 테스트 코드의 역할을 구분한다.

학생이 확인해야 할 것:
- `README.md`
- `docs/중요_시뮬레이터_개발_전달_명세.md`
- `docs/TDD_개발로드맵_학생가이드.md`
- `tests/unit/*`
- `tests/integration/*`

완료 기준:
- `npm install`
- `npm run verify`

이 단계의 산출물:
- 프로젝트 구조 이해
- 테스트와 구현의 관계 이해

### 3.2 1단계. 순수 계산 로직 확장
목표:
- 온도, 진동, 상태 전이처럼 외부 의존성이 없는 로직을 먼저 완성한다.

학생이 해야 할 일:
1. 새 규칙에 대한 단위 테스트를 먼저 작성한다.
2. 테스트 실패를 확인한다.
3. 최소 구현으로 테스트를 통과시킨다.
4. 리팩터링한다.

예시 대상:
- 진동 센서값 계산
- 기준 온도 복귀 보정
- 발행 payload 최소 계약
- 무효 제어값 처리 규칙

완료 기준:
- `npm run test:unit`

이 단계의 산출물:
- 순수 함수 중심 도메인 로직
- 규칙별 단위 테스트

### 3.3 2단계. 메시지 계약과 파서 완성
목표:
- MQTT payload 규약과 실제 코드 해석 규칙을 일치시킨다.

학생이 해야 할 일:
1. 제어 메시지/발행 메시지 계약을 문서에 명확히 적는다.
2. 메시지 파서 테스트를 먼저 작성한다.
3. 잘못된 payload에 대한 방어 로직을 구현한다.

예시 대상:
- `power=on/off` 외 값 무시
- 필수 필드 누락 처리
- 잘못된 JSON 수신 시 상태 불변

완료 기준:
- 관련 단위 테스트 전부 녹색
- 문서와 코드가 충돌하지 않음

이 단계의 산출물:
- 메시지 계약 문서
- 파서 테스트
- 방어 로직

### 3.4 3단계. 시뮬레이터 흐름 완성
목표:
- 제어 메시지 수신, 내부 상태 변경, tick 기반 계산, 토픽 발행까지 전체 흐름을 완성한다.

학생이 해야 할 일:
1. 통합 테스트를 먼저 작성한다.
2. 브로커 연결, 구독, 발행 흐름을 구현한다.
3. 정기 발행과 control 직후 즉시 발행 규칙을 검증한다.

예시 대상:
- 10 tick 누적 후 4개 topic 정기 발행
- control 후 해당 status 즉시 발행
- 10 tick 후 온도 변화 반영
- 컨베이어 상태와 진동 발행 연동

완료 기준:
- `npm run test:integration`

이 단계의 산출물:
- 동작하는 시뮬레이터 본체
- MQTT 기반 통합 테스트

### 3.5 4단계. 실행 엔트리포인트와 운영 모드 추가
목표:
- 실제 MQTT 브로커에 연결해 실행 가능한 형태로 만든다.

학생이 해야 할 일:
1. 실행용 엔트리포인트를 추가한다.
2. 브로커 주소, tick 설정 등 런타임 설정을 외부화한다.
3. 시작/종료 절차를 정리한다.

예시 대상:
- `npm run dev`
- `npm start`
- `.env.example` 기반 환경변수 설정
- 시작 후 자동 scheduler와 10초 정기 발행 검증

완료 기준:
- 수동 실행 가능
- 주요 로그 확인 가능
- build/test/integration과 충돌하지 않음

이 단계의 산출물:
- 실제 실행 가능한 프로그램
- README 실행 안내

### 3.6 5단계. MVP 완료 정리
목표:
- 코드, 테스트, 문서가 같은 내용을 말하도록 정리한다.

학생이 해야 할 일:
1. 테스트 케이스 누락이 없는지 확인한다.
2. 설계 문서와 상수 정의가 일치하는지 확인한다.
3. 최종 게이트를 통과한다.

완료 기준:
- `npm run verify`
- 문서 최신화
- MVP 범위 내 요구사항 충족

이 단계의 산출물:
- 첫 번째 최종 결과물

### 3.7 6단계. Node-RED Flow 연동 준비
목표:
- Node-RED import JSON을 GitHub에서 함께 관리한다.
- 시뮬레이터와 Node-RED가 같은 MQTT 토픽 계약을 사용하도록 한다.

학생이 해야 할 일:
1. `node-red/flows/` 아래에 import JSON을 추가한다.
2. `{uniq-user-id}`와 `room-01` 기준 토픽이 명세와 일치하는지 확인한다.
3. 시뮬레이터 실행 후 Node-RED에서 센서/상태 메시지가 수신되는지 확인한다.

권장 파일명:
- `node-red/flows/시트1.센서데이터수신및인리치먼트.json`
- `node-red/flows/시트2.룰엔진.json`
- `node-red/flows/시트3.AI에이전트-현장분석가.json`
- `node-red/flows/시트4.AI에이전트-관리자.json`
- `node-red/flows/시트5.2D대시보드.json`

완료 기준:
- Node-RED에서 JSON import 성공
- `kiot/{uniq-user-id}/factory/room-01/#` 기준 메시지 흐름 확인
- AI 에이전트 flow 사용 시 Node-RED 실행 환경에 `GEMINI_API_KEY` 설정

## 4. 운영 안정화 확장 단계
MVP 이후에는 아래 2단계를 추가로 진행할 수 있다.

### 4.1 7단계. 내구성 강화
- 브로커 재연결
- 잘못된 메시지 반복 수신
- 장시간 실행 안정성
- 장시간 발행 누락 여부 검증

### 4.2 8단계. 릴리즈 정리
- README 실행 안내 정리
- 데모 시나리오 작성
- 운영 체크리스트 작성
- 수용 기준 문서화

## 5. 학생용 반복 사이클
각 단계는 아래 사이클로 반복한다.

1. 요구사항을 작은 Slice로 자른다.
2. 테스트를 먼저 작성한다.
3. 테스트 실패를 확인한다.
4. 최소 구현으로 통과시킨다.
5. 리팩터링한다.
6. 관련 테스트를 다시 실행한다.
7. 기능 단위 종료 시 `npm run verify`를 실행한다.
8. 문서를 최신 상태로 갱신한다.

## 6. 학생이 자주 놓치는 포인트
- 테스트 실패를 실제로 확인하지 않고 구현부터 시작하는 것
- 숫자 리터럴을 로직 안에 직접 넣는 것
- 문서를 안 바꾸고 코드만 고치는 것
- 단위 테스트만 통과시키고 통합 테스트를 생략하는 것
- 완료 선언 전에 `npm run verify`를 생략하는 것

## 7. 학생 실습용 체크리스트
### 7.1 시작 전
- 설계 문서를 읽었는가
- 현재 테스트 범위를 이해했는가
- 이번 Slice의 완료 기준을 한 문장으로 말할 수 있는가

### 7.2 구현 중
- 테스트를 먼저 작성했는가
- 실패를 확인했는가
- 최소 코드만 작성했는가
- 상수를 분리했는가

### 7.3 종료 전
- `npm run build`
- `npm run test:unit`
- `npm run test:integration`
- `npm run verify`
- 문서 최신화

## 8. 최종 이해 포인트
이 프로젝트는 단순히 코드를 완성하는 실습이 아니다.
학생은 아래 3가지를 함께 훈련해야 한다.

- 설계 문서를 코드와 동기화하는 법
- 테스트를 먼저 작성하며 기능을 설계하는 법
- MQTT와 시간 기반 시뮬레이션을 검증 가능한 구조로 만드는 법
