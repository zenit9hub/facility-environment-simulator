# 시뮬레이터 개발 전달 명세

## 문서 목적

- 이 문서는 공장 시뮬레이터 실제 개발 담당자에게 전달하는 기준 문서입니다.
- `06. 구현 상세/01. 시뮬레이터 및 노드레드 시트별 토픽 명세` 폴더의 단일 기준 문서입니다.
- 범위는 `Node-RED와 문제없이 연동되기 위한 시뮬레이터 측 계약`입니다.
- 시뮬레이터 내부 로직 전체를 강하게 제한하지는 않지만, 강의용 기준으로 고정할 발행 주기와 온도 변화 모델 상수는 이 문서에서 명시합니다.

## 0. MQTT 네임스페이스와 운영 규칙

### 기본 구조

- Level 1: `kiot`
- Level 2: `{uniq-user-id}`
- Level 3: `factory` 또는 `dt`
- Level 4 이후: `factory/room-01/{domain}/{type}`

브로커 기준:

- 기본 MQTT URL: `mqtt://broker.emqx.io:1883`
- `MQTT_BROKER_URL` 환경변수로 브로커를 주입할 수 있습니다.
- 시뮬레이터, Node-RED, MQTTX는 동일한 브로커를 사용합니다.
- 강사가 원격으로 수강생 토픽을 직접 확인해야 하므로 로컬 브로커보다 공용 브로커를 기본값으로 둡니다.

prefix:

- 실세계 팩토리: `kiot/{uniq-user-id}/factory`
- 디지털 트윈: `kiot/{uniq-user-id}/dt/factory`

운영 규칙:

- 강사는 `kiot/#`, `kiot/+/factory/#`, `kiot/+/dt/factory/#`를 구독해 전체 수강생 흐름을 관제합니다.
- 수강생은 자신의 `uniq-user-id` 하위 토픽만 발행하고 구독합니다.
- 시뮬레이터는 `factory/...`만 발행/구독합니다.
- Node-RED 시트 1은 `factory -> dt/factory` 변환을 담당합니다.
- 룰 엔진만 실제 `control` 토픽을 발행합니다.
- AI는 직접 제어하지 않고 `analysis`, `ops/recommendation`만 담당합니다.

## 1. 현재 구현 대비 변경점

현재 구현과 비교했을 때, 아래 항목은 반드시 맞춰야 합니다.

- 토픽 구조 변경
  - 현재: `factory/.../conveyor-01/...`, `temp-01`, `vib-01`
  - 변경: `kiot/{uniq-user-id}/factory/room-01/...`
- 발행 payload 단순화
  - 센서: `{ "value": number }`
  - 상태: `{ "power": "on" | "off" }`
- 제어 payload 단순화
  - 공통: `{ "power": "on" | "off", "reason": "..." }`
  - 단, `reason`은 선택입니다.
- `aircon/status` 발행 추가
- 브로커 기준 변경
  - 현재 기본값: `127.0.0.1`
  - 변경 기준: `mqtt://broker.emqx.io:1883`

## 2. 시뮬레이터 측 필수 계약

Node-RED와 상호 연동되기 위해 시뮬레이터가 맞춰야 하는 최소 조건은 아래입니다.

 - MQTT 브로커 기본값: `mqtt://broker.emqx.io:1883`
- `uniq-user-id`는 `MQTT_UNIQ_USER_ID` 환경변수로 받습니다.
- 브로커는 `MQTT_BROKER_URL` 환경변수로 받을 수 있습니다.
- `roomId`는 현재 기준 `room-01`을 사용합니다.
- 아래 발행 토픽 4개를 발행합니다.
- 아래 구독 토픽 2개를 구독합니다.
- 최소 payload 규약을 맞춥니다.

즉, 토픽 계약과 payload 규약 외 세부 구현은 시뮬레이터 내부 구현 영역입니다. 단, 아래에 명시한 발행 주기와 온도 변화 기준값은 강의용 설계 상수로 고정합니다.

## 3. 환경변수 기준

### 필수

- `MQTT_UNIQ_USER_ID`
  - 예: `student-01`
  - 모든 토픽 경로의 `{uniq-user-id}` 자리에 들어갑니다.
  - 누락 시 시뮬레이터는 시작하지 않고 `fail-fast` 처리합니다.

### 선택

- `MQTT_BROKER_URL`
  - 예: `mqtt://broker.emqx.io:1883`
  - 설정 시 전체 MQTT URL 형태로 받습니다.
  - 미설정 시 기본값은 `mqtt://broker.emqx.io:1883`입니다.

### 현재 기준 고정값

- roomId: `room-01`

## 3-1. 시뮬레이터 내부 기본 상수

- `PUBLISH_INTERVAL_MS = 10000`
- `AIRCON_MIN_TEMP = 22`
- `AMBIENT_BASELINE_TEMP = 25`
- `CONVEYOR_MAX_TEMP = 50`
- `CONVEYOR_HEAT_DELTA = +0.2`
- `CONVEYOR_OVERHEAT_MODE = off`
- `CONVEYOR_OVERHEAT_EXTRA_DELTA = +1.0`
- `AIRCON_COOL_DELTA = -0.3`
- `AMBIENT_RECOVERY_DELTA = 0.1`

규칙:

- 센서값과 actuator status 값은 기본 10초 주기로 발행합니다.
- 발행 주기는 상수값으로 두어 필요 시 변경 가능하게 구현합니다.
- 기본 설계 기준은 10초입니다.
- 에어컨만 켜고 방치할 경우 온도는 `22도`까지 내려가고 그 이하로는 내려가지 않습니다.
- 컨베이어벨트만 가동하고 냉각이 없을 경우 온도는 `50도`까지 올라가고 그 이상으로는 올라가지 않습니다.
- 두 장비가 모두 꺼져 있으면 온도는 `25도`를 향해 복귀합니다.
- 컨베이어벨트가 동작 중이더라도 에어컨이 함께 켜지면 냉각 효과가 더 크게 작동하도록 구현합니다.
- `컨베이어 벨트 과열 모드`는 기본 `off`이며, `on`일 때는 컨베이어 가열량에 `+1.0도`를 추가합니다.
- 과열 모드는 별도 simulator 전용 토픽으로 분리하지 않고 `conveyor-belt/control` payload 안의 `overheatMode`로 함께 제어합니다.
- 이 모드는 냉각 실패, 워닝, 셧다운 시나리오를 빠르게 재현하기 위한 테스트용 설계 옵션입니다.

## 3-2. 온도 변화 규칙

온도 변화 단위는 `PUBLISH_INTERVAL_MS` 기준으로 적용합니다. 현재 기준에서는 `10초마다` 한 번씩 아래 규칙을 적용합니다.

- 컨베이어벨트 `on`, 에어컨 `off`
  - 온도는 tick당 `+0.2도`씩 상승합니다.
  - 최대 `50도`까지만 올라갑니다.
- 컨베이어벨트 `off`, 에어컨 `on`
  - 온도는 tick당 `-0.3도`씩 하강합니다.
  - 최저 `22도`까지만 내려갑니다.
- 컨베이어벨트 `off`, 에어컨 `off`
  - 온도는 평상 온도 `25도`를 향해 tick당 `0.1도`씩 복귀합니다.
  - 현재 온도가 `25도`보다 높으면 `-0.1도`, 낮으면 `+0.1도`로 이동합니다.
- 컨베이어벨트 `on`, 에어컨 `on`
  - 온도는 tick당 `-0.1도`씩 하강합니다.
  - 설명: 컨베이어벨트의 가열 `+0.2도`보다 에어컨의 냉각 `-0.3도`가 커서 순변화는 `-0.1도`입니다.
  - 최저 `22도` 아래로는 내려가지 않습니다.

## 3-3. 컨베이어 벨트 과열 모드

과열 모드는 기본 동작을 깨지 않으면서, 고도화 테스트용으로 급격한 온도 상승을 재현하기 위한 옵션입니다.

- 기본값은 `off`입니다.
- `on`일 때, 컨베이어벨트가 동작 중이면 기존 가열량 `+0.2도`에 `+1.0도`를 추가합니다.
- `power=off` 상태에서는 `overheatMode`가 `on`이어도 발열 계산에는 참여하지 않습니다.
- 즉, 과열 모드 `on` + 컨베이어벨트 `on` + 에어컨 `off`이면 tick당 `+1.2도` 상승합니다.
- 과열 모드 `on` + 컨베이어벨트 `on` + 에어컨 `on`이면 순변화는 `+0.9도`입니다.
  - 설명: 기본 가열 `+0.2도` + 과열 추가 `+1.0도` - 에어컨 냉각 `0.3도` = `+0.9도`
- 최대 온도 상한은 기존과 동일하게 `50도`입니다.

운영 의도:

- 냉각 기능이 켜져 있어도 발열이 잡히지 않는 실패 시나리오를 재현할 수 있습니다.
- `warning -> critical -> shutdown` 흐름을 짧은 시간 안에 검증할 수 있습니다.
- 강의와 테스트에서 급격한 온도 상승을 빠르게 관찰하기 좋습니다.

## 6. payload 최소 규약

### 센서 payload

```json
{
  "value": 31.2
}
```

```json
{
  "value": 0.23
}
```

규칙:

- `value`는 필수입니다.
- temperature와 vibration 모두 같은 형태를 사용합니다.

### 액추에이터 상태 payload

```json
{
  "power": "on",
  "overheatMode": "off"
}
```

```json
{
  "power": "off",
  "overheatMode": "on"
}
```

규칙:

- `power`는 필수입니다.
- 값은 `on` 또는 `off`만 사용합니다.
- `overheatMode`는 선택입니다.
- `overheatMode`가 있으면 값은 `on` 또는 `off`만 사용합니다.
- 컨베이어벨트 status에는 현재 `power`와 함께 `overheatMode`를 같이 담는 편이 좋습니다.
- status 발행 시에는 현재 컨베이어 상태를 명확히 보여주기 위해 `power`와 `overheatMode`를 함께 담는 방식을 권장합니다.

### 액추에이터 제어 수신 payload

```json
{
  "power": "off"
}
```

```json
{
  "power": "off",
  "overheatMode": "on",
  "reason": "rule-critical"
}
```

규칙:

- `power`는 필수입니다.
- `reason`은 선택입니다.
- 에어컨 제어는 `power`와 `reason`만 사용합니다.
- 컨베이어벨트 제어는 `overheatMode`를 추가 필드로 함께 사용할 수 있습니다.
- `overheatMode`는 선택이며 값은 `on` 또는 `off`만 사용합니다.
- `overheatMode`가 payload에 없으면 현재 상태를 유지합니다.
- `overheatMode`를 받으면 정기 발행 주기를 기다리지 않고 `conveyor-belt/status`를 즉시 추가 발행합니다.
- 강의용 테스트와 수동 제어에서는 `power`와 `overheatMode`를 함께 보내 전체 상태를 명시하는 방식을 권장합니다.

## 7. 시뮬레이터가 해야 할 최소 동작

- 시작 후 센서 토픽과 상태 토픽을 발행할 수 있어야 합니다.
- 온도, 진동, 컨베이어 status, 에어컨 status는 기본 10초 주기로 발행합니다.
- `conveyor-belt/control`을 받으면 내부 상태를 반영한 뒤 `conveyor-belt/status`를 즉시 추가 발행합니다.
- `aircon/control`을 받으면 내부 상태를 반영한 뒤 `aircon/status`를 즉시 추가 발행합니다.
- `aircon/status` 발행은 필수입니다.
- `conveyor-belt/control`에서 `overheatMode`를 받으면 `conveyor-belt/status`에 함께 반영해 즉시 추가 발행합니다.
- `MQTT_UNIQ_USER_ID`가 없으면 즉시 종료합니다.
- 브로커 연결 시 런타임 URL은 `MQTT_BROKER_URL` 또는 기본값 `mqtt://broker.emqx.io:1883`를 사용합니다.

정리:

- 정기 발행은 10초 단위입니다.
- actuator control이 발생한 경우에는 정기 주기를 기다리지 않고 status를 즉시 추가 발행합니다.
- 온도 하한은 `22도`, 평상 복귀 온도는 `25도`, 상한은 `50도`입니다.
- 온도 변화량은 컨베이어 `+0.2도`, 에어컨 `-0.3도`, 평상 복귀 `0.1도`를 기준으로 합니다.

## 8. Node-RED 관점에서 실제로 필요한 것

Node-RED는 아래만 맞으면 됩니다.

- 같은 브로커를 사용한다.
- 같은 `{uniq-user-id}`와 `room-01`을 사용한다.
- 발행 토픽 4개와 구독 토픽 2개가 일치한다.
- 센서는 `value`, 상태와 제어는 `power`를 포함한다.

그 외 내부 시뮬레이션 방식은 시뮬레이터 구현 담당자 재량입니다.

## 9. 콘솔 로그 최소 규약

로그는 MQTT 토픽 계약이 아니라 교육과 디버깅을 돕기 위한 콘솔 출력입니다. 너무 많은 로그를 남기기보다, 정기 발행과 제어 수신이 확인될 정도로만 간략히 출력합니다.

정기 발행 로그:

```text
[simulator] MQTT topic messages published. intervalMs=10000 userId=zenit roomId=room-01
```

제어 메시지 수신 로그:

```text
[simulator] control message received. target=aircon power=on reason=manual-test
```

```text
[simulator] control message received. target=conveyor-belt power=on overheatMode=off reason=manual-test
```

즉시 status 발행 로그:

```text
[simulator] actuator status published immediately. target=aircon power=on
```

```text
[simulator] actuator status published immediately. target=conveyor-belt power=on overheatMode=off
```

규칙:

- 정기 발행 로그는 `PUBLISH_INTERVAL_MS` 주기마다 한 줄만 남깁니다.
- 제어 메시지를 수신하면 대상 장비, `power`, 선택값 `reason`을 남깁니다.
- 컨베이어벨트 제어 메시지에 `overheatMode`가 있으면 함께 남깁니다.
- 제어 수신 후 즉시 status를 추가 발행한 경우, 즉시 발행 로그를 한 줄 남깁니다.
- payload 전체를 매번 길게 출력하기보다 수강생이 확인해야 할 핵심 필드만 출력합니다.

## 10. 구현 시 강조 포인트

- 시뮬레이터와 Node-RED는 토픽 기준으로 명확히 분리합니다.
- 시트 1은 `factory -> dt/factory`를 담당하는 첫 번째 의미 변환 단계입니다.
- 시트 1은 per-topic DT 토픽 외에 `state/current` 스냅샷도 함께 생성합니다.
- 시트 2 룰 엔진은 여러 per-topic 메시지를 직접 조합하기보다 `state/current`를 1차 입력으로 사용합니다.
- 룰 엔진만 `control` 토픽을 발행합니다.
- AI 에이전트는 `analysis`와 `ops/recommendation`만 담당합니다.
- `warning` 상황에서는 에어컨을 먼저 가동하고, 이후에도 위험이 지속되면 컨베이어벨트를 정지합니다.
- 3D는 현재 단계에서 `dt/factory` 센서 및 액추에이터 상태를 소비합니다.
- 추후 고도화 시 3D에 `rule/result`, `ops/recommendation`을 추가할 수 있습니다.
