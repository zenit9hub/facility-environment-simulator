# TDD_운영사이클

## 1. 목적
이 문서는 설비환경시뮬레이터 개발 시 테스트 주도 개발의 공식 작업 순서를 정의한다.

## 2. 기본 원칙
- 구현보다 테스트가 먼저다.
- 테스트 실패를 확인하지 않은 구현은 허용하지 않는다.
- 단위 테스트가 먼저 녹색이 되어야 다음 단계로 간다.
- 기능 단위 개발 완료 판정은 통합 테스트 통과까지 포함한다.

## 3. 공식 작업 순서
### 3.1 Slice 단위 개발
1. 요구사항을 가장 작은 기능 단위로 자른다.
2. 해당 Slice의 기대 동작을 테스트 케이스로 먼저 작성한다.
3. 테스트를 실행해 실패를 확인한다.
4. 테스트를 통과시키는 최소 구현을 추가한다.
5. 중복 제거와 구조 개선을 위한 리팩터링을 수행한다.
6. 관련 단위 테스트를 다시 실행한다.

### 3.2 기능 완료 게이트
아래가 모두 통과되어야 해당 기능은 완료로 본다.

1. `npm run build`
2. `npm run test:unit`
3. `npm run test:integration`

권장 실행은 `npm run verify`다.

## 4. 테스트 분류 규칙
- 단위 테스트: 순수 계산, 상태 전이, 파서, 발행 payload 최소 계약
- 통합 테스트: MQTT 연결, 제어 메시지 수신, tick 발행, 토픽/페이로드 연동

## 5. 개발 완료 정의
다음 조건이 충족되면 개발 완료로 판정한다.

- 요구된 테스트 케이스가 모두 구현되어 있다.
- 구현 대상 기능이 모든 단위 테스트를 통과한다.
- 시스템 흐름이 통합 테스트를 통과한다.
- 설계 문서와 코드 상수 정의가 일치한다.

## 6. 금지 사항
- 구현 후 테스트를 덧붙이는 방식
- 실패 확인 없이 테스트를 작성한 것으로 간주하는 방식
- 통합 테스트 미실행 상태에서 기능 완료 선언
- 상수 정의 없이 숫자 리터럴을 직접 사용하는 방식
