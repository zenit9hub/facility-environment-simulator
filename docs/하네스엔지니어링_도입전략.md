# 하네스엔지니어링_도입전략

## 1. 이 프로젝트에서의 하네스 엔지니어링 정의
이 프로젝트에서 하네스 엔지니어링은 단순 테스트 도구 도입이 아니다.
개발이 흔들리지 않도록 아래를 묶은 운영 구조를 뜻한다.

- 설계 문서
- 메시지/상수 규약
- 테스트 하네스
- 자동 검증 명령
- 완료 게이트

즉, 구현 코드 바깥에서 개발 품질을 강제하는 레일 전체를 하네스로 본다.

## 2. 왜 필요한가
설비환경시뮬레이터는 아래 특성이 있다.

- 외부 입력이 MQTT 메시지다.
- 상태 변화가 시간 기반이다.
- 출력이 다중 토픽 발행이다.
- 계산 규칙이 상수 조정에 민감하다.

이런 종류의 시스템은 화면이 없어도 버그가 숨어들기 쉽다.
따라서 코드보다 먼저 검증 구조를 두는 편이 안전하다.

## 3. 이 프로젝트에 적용할 하네스 구성 요소
### 3.1 Context Engineering
- 설계 담당자 전달 명세
- 테스트 전략 문서

### 3.2 Tool Orchestration
- `npm run test:unit`
- `npm run test:integration`
- `npm run verify`

### 3.3 State And Memory
- 상수 전용 파일
- 시뮬레이터 상태 스냅샷 조회 함수
- 최소 상태 유지 원칙

### 3.4 Verification
- 단위 테스트
- MQTT 통합 테스트
- TypeScript build 통과
- 필요 시 외부 테스트 브로커 기반 smoke/integration 검증

### 3.5 Human Gate
- 설계 문서 변경 시 관련 테스트 함께 갱신
- 메시지 규약 변경 시 통합 테스트 먼저 갱신
- 완료 선언 전 `npm run verify` 필수

### 3.6 Lifecycle
- 시뮬레이터 start/stop 규약
- 연결 실패와 메시지 파싱 실패 처리 원칙
- 테스트에서 독립 브로커를 띄우고 종료하는 하네스

## 4. 반드시 문서화해야 하는 항목
- MQTT 토픽 목록
- 제어 메시지 payload 규약
- 상태값/센서값 payload 규약
- 상수 목록과 기본값
- 온도/진동 계산 규칙
- 발행 payload 최소 필드 규약
- 테스트 케이스 목록
- 기능 완료 정의

## 6. 운영 원칙
- 새로운 기능은 테스트가 먼저다.
- 비즈니스 규칙은 상수와 순수 함수로 먼저 분리한다.
- MQTT 연동은 반드시 통합 테스트로 검증한다.
- 테스트가 어려운 구조는 실제 구현 전에 구조를 다시 나눈다.
- 로컬 검증은 코드 완결성 확인용으로 충분히 활용하되, 실제 브로커 연동 검증은 외부 테스트 브로커를 우선할 수 있다.
